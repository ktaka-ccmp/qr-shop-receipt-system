# 電子帳簿保存法関連 特許調査レポート

**作成日:** 2026年2月4日
**調査対象:** タイムスタンプ、訂正削除履歴、監査証跡に関する特許

---

## 1. 調査概要

### 1.1 調査目的

電子領収書発行システムにおいて、電子帳簿保存法対応機能（タイムスタンプ付与、訂正削除履歴の保持、監査証跡の記録）を実装する際の特許侵害リスクを評価する。

### 1.2 調査方法

- Web検索による公開情報の収集
- 関連技術の標準規格・認定制度の調査
- 既存サービス・製品の特許状況の調査

### 1.3 主要な発見

| 領域 | 特許リスク | 評価 |
|------|-----------|------|
| タイムスタンプ技術 | **低** | 国際標準規格（RFC3161）に準拠した実装は安全 |
| 訂正削除履歴管理 | **低〜中** | 一般的なデータベース設計手法、特定の特許は未発見 |
| 監査証跡（監査ログ） | **低** | 一般的なログ管理技術、特定の特許は未発見 |
| 自動仕訳・勘定科目分類 | **高** | freee特許5503795（キーワード対応テーブル方式）に注意 |

## 2. タイムスタンプ技術

### 2.1 技術概要

タイムスタンプは、電子データが特定の時刻に存在し、その時点以降改ざんされていないことを証明する技術である。

**基本的な仕組み:**
1. 電子データからハッシュ値を計算
2. ハッシュ値をタイムスタンプ局（TSA）に送信
3. TSAが時刻情報とハッシュ値を結合し、電子署名を付与
4. 署名済みタイムスタンプトークンを発行

### 2.2 標準規格

| 規格 | 説明 | 状況 |
|------|------|------|
| RFC3161 | インターネット標準のタイムスタンプ規格 | 広く普及、オープン標準 |
| RFC5816 | RFC3161の更新版（ESSCertIDv2対応） | 現行標準 |
| ISO/IEC 18014 | 国際標準規格 | JIS X5063として日本でも標準化 |
| JIS X5094:2011 | UTCトレーサビリティ保証のためのTAA技術要件 | 日本発の世界初の標準 |

### 2.3 認定タイムスタンプ事業者（日本）

総務大臣認定の時刻認証業務認定事業者:

| 事業者 | サービス名 | 認定取得 |
|--------|-----------|---------|
| アマノ株式会社 | アマノタイムスタンプサービス3161 | 2023年2月 |
| セイコーソリューションズ株式会社 | セイコータイムスタンプサービス | 2023年2月 |
| 三菱電機インフォメーションネットワーク株式会社 | MINDタイムスタンプサービス | 2023年2月 |
| 株式会社TKC | TKCタイムスタンプサービス | - |
| 株式会社サイバーリンクス | CloudCerts | - |
| GMOグローバルサイン株式会社 | 認定タイムスタンプ byGMO | 2024年3月（新規参入第一号） |

### 2.4 特許状況

**調査結果:** タイムスタンプの基本技術に関する特許は、主に**標準規格（RFC3161等）に基づいており、オープンな技術**として広く利用可能。

**リスク評価: 低**

- RFC3161準拠のタイムスタンプ実装は、標準規格に則った一般的な技術
- 認定タイムスタンプ事業者のAPIを利用する場合、特許リスクは当該事業者側が負担
- 独自のタイムスタンプ局を構築する場合は、追加調査を推奨

**推奨:**
- 認定タイムスタンプ事業者のAPIを利用（アマノ、セイコー、GMO等）
- RFC3161準拠の実装を行う
- 独自の時刻認証技術を開発しない

## 3. 訂正削除履歴管理

### 3.1 電子帳簿保存法の要件

電子帳簿保存法において、電子取引データの保存には「真実性の確保」が求められる。以下のいずれかの措置が必要:

1. タイムスタンプが付与されたデータを受領
2. 受領後遅滞なくタイムスタンプを付与
3. **データの訂正削除を行った場合にその記録が残るシステム**を利用
4. **訂正削除ができないシステム**を利用
5. 訂正削除の防止に関する事務処理規程を策定・運用

### 3.2 技術的実装方法

**方法1: 訂正削除履歴が残るシステム**
```
[元データ] → [訂正] → [訂正後データ] + [訂正履歴レコード]
                        ├── 訂正前の値
                        ├── 訂正後の値
                        ├── 訂正日時
                        └── 訂正者情報
```

**方法2: 訂正削除ができないシステム（イミュータブル）**
```
[データ] → 物理的に削除・変更不可
          └── 論理削除のみ可能（フラグ付与）
```

### 3.3 特許状況

**調査結果:** 訂正削除履歴管理に関する**特定の支配的な特許は発見されなかった**。

**理由:**
- データベースの履歴管理は一般的な設計パターン
- 監査ログの記録は広く知られた技術
- イミュータブルデータベースはオープンソースで複数実装が存在

**リスク評価: 低〜中**

**発見された関連技術:**

| 技術 | 特許状況 | 備考 |
|------|---------|------|
| ブロックチェーン証明 | サイバーリンクス「CloudCerts」特許取得 | ブロックチェーン+証明書の組み合わせ |
| メタデータ改ざん検知 | JP5350782B2（Google Patents） | ハッシュ値による検知方式 |

**推奨:**
- 一般的なデータベース設計パターンを採用（履歴テーブル方式）
- ブロックチェーンを使用する場合は、CloudCertsとの差別化を確認
- JIIMA認証取得済みシステムの設計を参考にする

## 4. 監査証跡（監査ログ）

### 4.1 概要

監査証跡とは、システム上で「いつ」「誰が」「どのような処理を行ったか」を時系列で記録したログデータ。

**記録すべき情報:**
- タイムスタンプ（処理日時）
- 操作者情報（ユーザーID、IPアドレス等）
- 操作内容（作成/参照/更新/削除）
- 対象データの識別子
- 変更前後の値（更新の場合）

### 4.2 特許状況

**調査結果:** 監査ログ管理に関する**特定の支配的な特許は発見されなかった**。

**リスク評価: 低**

**理由:**
- ログ管理は広く知られた一般的技術
- 多数のオープンソースソリューションが存在
- クラウドサービス（AWS、Azure、GCP）で標準機能として提供

**推奨:**
- 一般的なログ管理パターンを採用
- クラウドサービスの監査ログ機能を活用（AWS CloudTrail、GCP Cloud Audit Logs等）
- 独自の高度な監査技術を開発する場合は追加調査

## 5. 長期署名（PAdES/XAdES）

### 5.1 概要

長期署名は、電子署名の有効性を長期間にわたり保証するフォーマット。電子帳簿保存法の10年保存要件への対応に有用。

| 規格 | 対象 | 説明 |
|------|------|------|
| PAdES | PDF文書 | PDF内に署名・タイムスタンプ・検証情報を内包 |
| XAdES | XML文書 | 大量文書の長期保管に適する |
| CAdES | CMS形式 | バイナリ署名 |

### 5.2 特許状況

**調査結果:** PAdES/XAdESは**ETSI（欧州電気通信標準化機構）により標準化されたオープン規格**。

**リスク評価: 低**

**主要な製品・サービス:**
- アンテナハウス「長期署名PAdES/XAdESライブラリ」
- セイコーソリューションズ「eviDaemon」
- PFU「PDF長期署名ライブラリ」（2024年4月販売終了）
- サイバートラスト「iTrustリモート署名サービス」

**推奨:**
- 認定タイムスタンプ事業者の長期署名サービスを利用
- 標準規格（ETSI EN 319 132等）に準拠した実装

## 6. 会計ソフト連携時の注意点

### 6.1 freee特許（再掲）

**特許第5503795号（自動仕訳）**

| 項目 | 内容 |
|------|------|
| 権利者 | フリー株式会社 |
| 技術内容 | キーワード対応テーブルによる勘定科目自動分類 |
| 判例 | freee vs マネーフォワード訴訟（2017年、freee敗訴） |

**回避策:**
- キーワード対応テーブル方式を使用しない
- 機械学習方式は非侵害（判例あり）
- 勘定科目の自動判定機能自体を実装しない

### 6.2 会計ソフト連携の安全な実装

**安全な実装:**
- 仕訳データのCSV/API出力（形式変換のみ）
- 勘定科目はユーザーが手動選択
- freee/マネーフォワード等のAPIを呼び出すだけ

**リスクのある実装:**
- 取引内容からの勘定科目自動推定
- キーワードテーブルによる科目マッピング

## 7. JIIMA認証について

### 7.1 概要

JIIMA（公益社団法人日本文書情報マネジメント協会）が、電子帳簿保存法の要件を満たすソフトウェアを認証する制度。

**認証の種類:**
1. 電子帳簿ソフト法的要件認証（「優良な電子帳簿」対応）
2. 電帳法スキャナ保存ソフト法的要件認証
3. 電子書類ソフト法的要件認証
4. 電子取引ソフト法的要件認証

### 7.2 特許との関係

**注意:** JIIMA認証は「電子帳簿保存法の法的要件を満たすこと」を認証するものであり、**特許侵害がないことを保証するものではない**。

## 8. 結論と推奨事項

### 8.1 リスク評価まとめ

| 機能領域 | 特許リスク | 推奨アプローチ |
|---------|-----------|---------------|
| タイムスタンプ付与 | **低** | 認定事業者のAPIを利用 |
| 訂正削除履歴管理 | **低〜中** | 一般的なDB設計パターンを採用 |
| 監査証跡（ログ） | **低** | 標準的なログ管理を実装 |
| 長期署名（PAdES/XAdES） | **低** | 認定事業者のサービスを利用 |
| 勘定科目自動分類 | **高** | 実装しない、または機械学習方式 |

### 8.2 安全な実装方針

1. **タイムスタンプ:** 総務大臣認定の事業者（アマノ、セイコー、GMO等）のAPIを利用
2. **訂正削除履歴:** 履歴テーブル方式の一般的なDB設計を採用
3. **監査証跡:** 標準的なログ管理（誰が・いつ・何を・どうしたか）
4. **会計連携:** 仕訳データのエクスポートのみ、自動科目判定は非実装

### 8.3 追加調査推奨事項

| 項目 | 優先度 | 理由 |
|------|--------|------|
| サイバーリンクス CloudCerts特許詳細 | 中 | ブロックチェーン証明を実装する場合 |
| JIIMA認証取得システムの設計パターン | 低 | 参考として |
| クラウド会計ソフトの特許ポートフォリオ | 低 | 会計連携を拡充する場合 |
