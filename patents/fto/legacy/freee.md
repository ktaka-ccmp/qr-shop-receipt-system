## 3. 特許5503795 詳細分析

### 3.1 特許の概要

| 項目 | 内容 |
|------|------|
| **特許番号** | 特許5503795 |

### 3.2 請求項の要素分解

freee vs マネーフォワード訴訟における裁判所の整理に基づく：

```
【請求項1の構成要件】

A: ウェブサーバによるクラウドコンピューティング会計処理方法

B: ウェブ明細データを取引ごとに識別

C: 取引内容の記載に含まれうるキーワードと勘定科目との対応づけを保持する
   対応テーブルを参照して、特定の勘定科目に自動的に仕訳

D: 日付、取引内容、金額、勘定科目を含む仕訳データを作成

E: 複数のキーワードが含まれる場合にキーワードの優先ルールを適用し、
   優先順位の最も高いキーワードにより対応テーブルを参照
```

### 3.3 技術的特徴のポイント

**核心的特徴:**
1. **キーワード対応テーブル**: 特定のキーワードと勘定科目を予め対応付けたテーブル
2. **自動的な仕訳**: 人間の判断を介さず、システムが自動で勘定科目を決定
3. **優先ルール**: 複数キーワードがある場合の優先順位付け

**侵害となる処理フローの例:**

```
入力: 「JR五反田 モロゾフ」（店舗名）
  ↓
キーワード抽出: 「JR」「五反田」「モロゾフ」
  ↓
対応テーブル参照:
  - JR → 旅費交通費
  - 五反田 → 会議費
  - モロゾフ → 接待費
  ↓
優先ルール適用: モロゾフが最優先
  ↓
出力: 勘定科目「接待費」を自動付与

→ この処理フローは特許侵害の可能性が高い
```

### 3.4 権利範囲の限定

訴訟における審理を通じて、以下の点が権利範囲の限定要素として考えられます：

| 限定要素 | 説明 |
|----------|------|
| **対応テーブル** | 単純なルールベースでなく、テーブル形式での対応付けが必要 |
| **キーワード抽出** | 取引内容からキーワードを抽出するプロセスが必要 |
| **優先ルール** | 複数キーワード時の優先順位決定が必要 |
| **自動的な仕訳** | システムが自動で勘定科目を確定（候補提示のみは非該当の可能性） |

- 機械学習による勘定科目推測は、「キーワード対応テーブルを参照して自動的に仕訳」とは技術的に異なる

## 8. 侵害回避策の検討

### 8.1 回避策の比較表

| # | 回避策 | 侵害リスク | 実装難易度 | ユーザー体験 | 推奨度 |
|---|--------|-----------|-----------|-------------|--------|
| A | 勘定科目判定を行わない | **なし** | 低 | 普通 | ★★★★★ |
| B | 候補提示のみ（利用者が確定） | **低** | 中 | 良好 | ★★★★☆ |
| C | 機械学習方式 | **低** | 高 | 良好 | ★★★★☆ |
| D | 会計ソフト連携（判定は会計ソフト側） | **なし** | 中 | 良好 | ★★★★★ |
| E | 利用者履歴からの学習 | **要確認** | 中 | 良好 | ★★★☆☆ |
| F | 店舗業種からの推定のみ | **低** | 低 | 普通 | ★★★☆☆ |

### 8.2 推奨回避策の詳細

#### 回避策A: 勘定科目判定を行わない（最も安全）

```
処理フロー:
店舗 → 領収書データ生成 → サーバー保管 → 利用者ダウンロード
                                              ↓
                                        勘定科目は空欄のまま
                                              ↓
                                        会計ソフトで利用者が入力
```

**メリット:**
- 特許侵害リスクゼロ
- 実装が単純
- 会計ソフトの機能を活かせる

**デメリット:**
- ユーザーの手間が増える
- 競合との差別化要素が減る

#### 回避策B: 候補提示のみ（利用者が確定）

```
処理フロー:
店舗業種: 飲食店
  ↓
候補提示: 「会議費」「交際費」「福利厚生費」（選択肢として表示）
  ↓
利用者が選択・確定
  ↓
確定した勘定科目をデータに付与

→ 「自動的に仕訳」ではなく「候補提示」なので非侵害の可能性
```

**法的根拠:**
- freee特許は「自動的に仕訳」が必須要素
- 候補提示のみで利用者が確定する場合は「自動的」ではない

**リスク:**
- 「候補提示」が「自動的に仕訳」に該当するかは解釈の余地あり
- 弁理士による確認を推奨

#### 回避策C: 機械学習方式

```
処理フロー:
入力データ: 店舗名、業種、金額、日時、利用者属性
  ↓
機械学習モデル（ニューラルネットワーク等）
  ↓
勘定科目を推測・出力

→ freee vs マネーフォワード判決で非侵害と認定済み
```

**法的根拠:**
- 東京地裁判決（平成29年7月27日）でマネーフォワードの機械学習方式は非侵害と認定

**実装上の注意:**
- 機械学習モデルの設計・学習が必要
- 十分な学習データの確保が必要
- 「キーワード対応テーブル」を内部で使用しない設計とする

#### 回避策D: 会計ソフト連携（判定は会計ソフト側）

```
処理フロー:
本サービス: 構造化された領収書データを提供（勘定科目なし）
  ↓
API連携
  ↓
会計ソフト（freee、マネーフォワード等）: 勘定科目を判定

→ 本サービスは勘定科目判定を行わないため侵害なし
```

**メリット:**
- 特許侵害リスクゼロ
- 会計ソフトの高度な判定機能を活用可能
- 利用者は既存の会計ソフトでシームレスに利用可能

**連携候補:**
- freee会計
- マネーフォワードクラウド会計
- 弥生会計

### 8.3 回避策の組み合わせ（推奨）

**フェーズ別アプローチ:**

| フェーズ | 回避策 | 備考 |
|---------|--------|------|
| **MVP（フェーズ1）** | 回避策A（判定なし）+ 回避策D（会計ソフト連携） | 最小リスクでローンチ |
| **フェーズ2** | 回避策B（候補提示）の追加 | 弁理士確認後 |
| **フェーズ3** | 回避策C（機械学習）の追加 | 十分なデータ蓄積後 |
