# 電子領収書発行システム 特許侵害リスク分析（改訂版）

**作成日:** 2026年2月3日  
**目的:** 電子領収書発行システムの特許侵害リスク包括的調査

---

## 1. 重要な前提問題

### 1.1 「電子レシート」と「電子領収書」の区別について

**結論: 特許上は同一と見なされるリスクが高い**

| 観点 | 電子レシート | 電子領収書 | 特許上の評価 |
|------|-------------|-----------|------------|
| 法的性質 | 買い物明細 | 金銭受領の証拠 | **機能・構造が類似なら同一視される可能性** |
| 記載内容 | 商品明細、合計金額 | 金額、但し書き、宛名 | データ構造は類似 |
| 電子化方式 | QRコード/バーコード | QRコード | **技術的に同一** |
| 配信方式 | サーバー→端末 | サーバー→端末 | **技術的に同一** |

**東芝テックFAQの「領収書は紙で出力」との記載について:**
- これは**サービス上の区別**であり、特許クレームの解釈とは別問題
- 特許クレームは「電子レシート」という用語を使用しているが、技術的に同等の「電子領収書」システムにも及ぶ可能性

**弁理士確認必須事項:**
- [ ] 特許クレームにおける「電子レシート」の定義範囲
- [ ] 「電子領収書」が権利範囲に含まれるか否かの判断基準

---

### 1.2 QRコード表示方向の問題

**本システムのフロー（問題あり）:**
```
店舗がQRコード表示 → 利用者が読み取り → 利用者がQRコード表示 → 店舗が読み取り
         ↓                    ↓                    ↓                    ↓
    トークン取得          トークン取得完了        トークン提示          電子領収書発行
```

**東芝テック特許のフロー:**
```
利用者がバーコード/QRコード表示 → 店舗が読み取り → 電子レシート発行
```

**問題点:**
- 本システムには「**利用者がQRコード表示 → 店舗が読み取り**」の工程が含まれる
- この部分は東芝テック特許のフローと**同一**
- ワンタイムトークンの有無だけでは差別化が不十分な可能性

## 3. 特許6598936の詳細分析

### 3.1 請求項1の構成要素

```
【請求項1】
通信可能に接続された電子レシート管理サーバと、端末装置とを有する電子レシートシステムであって、

前記電子レシート管理サーバは、
(a) 決済対象の商品の取引内容を示す要素を少なくとも含んだ複数の要素で構成される
    電子レシート情報を、当該取引に係る消費者を識別する消費者コードと対応付けて
    記憶する記憶手段
(b) 前記消費者コードに対応付けられた前記端末装置に対し、前記消費者コードに
    対応付けて前記記憶手段に記憶されている前記電子レシート情報を出力する
    電子レシート出力手段

前記端末装置は、
(c) 前記電子レシート管理サーバから出力された前記電子レシート情報を受信する受信手段
(d) 前記受信手段が受信した前記電子レシート情報を表示するとともに、前記電子レシート
    情報を構成する所定の要素に関連付けてフラグが付加されている場合に、表示態様を
    変更して表示する表示手段
(e) 前記表示手段が表示した前記電子レシート情報に対する操作を受け付ける受付手段

前記表示手段は、前記表示態様を変更した前記電子レシート情報に対する操作を
前記受付手段が受け付けたことを条件に、当該電子レシート情報を変更前の
表示態様で表示させる
```

### 3.2 本システムとの比較

| 構成要素 | 特許6598936 | 本システム | 該当性 |
|---------|------------|-----------|--------|
| (a) 消費者コードとレシート対応付け | ○ | ワンタイムトークンと領収書対応付け | **類似** |
| (b) 消費者コード紐付け端末への出力 | ○ | トークン認証後の端末への出力 | **類似** |
| (c) 端末での受信 | ○ | ○ | **該当** |
| (d) フラグ付き要素の表示態様変更 | ○ | △（キャンペーンフラグは非実装予定） | **非該当の可能性** |
| (e) 操作受付 | ○ | ○（ただし表示態様変更なし） | **部分的** |

**評価:**
- 構成要素(d)の「フラグが付加されている場合に表示態様を変更」を実装しなければ、本特許クレームには**非該当の可能性**
- ただし、東芝テックは他の特許も保有しているため、包括的調査が必要

## 4. 電子レシート関連特許の包括的マッピング

### 4.1 主要方式と特許状況

| 方式 | 代表サービス | QRコード方向 | 事前登録 | 特許リスク |
|------|-------------|-------------|---------|-----------|
| **Type A: 利用者表示→店舗読取** | 東芝テック スマートレシート | 利用者→店舗 | 必要（会員登録） | **東芝テック特許該当** |
| **Type B: 店舗表示→利用者読取** | Airレジ、スマレジ | 店舗→利用者 | 不要 | 調査必要 |
| **Type C: メール/SMS送信** | Square | なし | メアド入力 | 低リスク |
| **Type D: 会員カード連携** | 各社ポイントカード | なし | 必要 | 調査必要 |

### 4.2 本システムの位置づけ

**本システム:** Type A + Type B の**ハイブリッド方式**

```
[Step 1] 店舗がQRコード表示 → 利用者が読み取り（Type B）
[Step 2] 利用者がQRコード表示 → 店舗が読み取り（Type A）← 問題の工程
[Step 3] 電子領収書発行
```

**リスク評価:**
- Step 2が東芝テック特許（Type A）と類似するフローを含む
- 「ワンタイムトークン」の使用だけでは差別化として不十分な可能性

## 6. 侵害回避のための設計変更案

### 6.1 案A: QRコードフロー変更（推奨）

**変更前:**
```
店舗表示 → 利用者読取 → 利用者表示 → 店舗読取 → 発行
```

**変更後（案A-1）: 店舗表示のみ**
```
店舗がQRコード表示（領収書データ含む/URLのみ）→ 利用者読取 → 電子領収書取得
```
- 利用者が何も表示しない
- Airレジ方式と類似
- 東芝テック特許の「消費者コード読取」フローを回避

**変更後（案A-2）: メールアドレス入力方式**
```
店舗で金額確定 → 利用者がメアド入力 → メールで電子領収書送信
```
- Square方式と類似
- QRコード読取フローを完全回避

### 6.2 案B: 技術的差別化の強調

東芝テック特許との差異を主張できる可能性のある要素:

| 差異点 | 東芝テック特許 | 本システム | 差別化の強度 |
|-------|--------------|-----------|-------------|
| コードの性質 | 固定の会員コード | ワンタイムトークン | **中**（クレーム解釈次第） |
| 対象文書 | レシート（買物明細） | 領収書（金銭受領証） | **弱**（機能が類似） |
| 事前登録 | 必要（アプリ会員登録） | 不要（都度認証） | **中** |
| 利用者端末 | 専用アプリ必須 | ブラウザのみ | **中** |

**評価:** 技術的差別化だけでは侵害回避の確実性が低い

## 9. 結論

### 9.1 リスク評価まとめ

| リスク領域 | レベル | 対策 |
|-----------|--------|------|
| **利用者QRコード表示→店舗読取** | **高** | フロー変更を強く推奨 |
| 電子レシート/領収書の区別 | **高** | 弁理士確認必須 |
| 勘定科目自動判定 | 中（回避可能） | 実装しない |
| キャンペーン機能 | 中（回避可能） | 実装しない |
| 返品処理連動 | 低（回避可能） | 実装しない |

### 9.2 最終推奨

1. **フロー変更を強く推奨**: 「利用者がQRコード表示→店舗が読取」の工程を削除し、「店舗がQRコード表示→利用者が読取」のみの方式に変更

2. **早期の弁理士相談**: 本分析は予備的調査であり、専門家による正式な特許鑑定が必須

3. **競合サービスの方式調査**: Airレジ、スマレジなど「店舗QR表示」方式のサービスの特許状況を追加調査
