## 2. 各特許の侵害リスク詳細分析

### 2.1 freee特許5503795（自動仕訳・キーワード対応テーブル）

#### 特許の技術的範囲

**請求項の要素分解（裁判所整理版）:**

| 構成要件 | 内容 |
|----------|------|
| A | ウェブサーバによるクラウドコンピューティング会計処理方法 |
| B | ウェブ明細データを取引ごとに識別 |
| C | **取引内容の記載に含まれうるキーワードと勘定科目との対応づけを保持する対応テーブルを参照して、特定の勘定科目に自動的に仕訳** |
| D | 日付、取引内容、金額、勘定科目を含む仕訳データを作成 |
| E | **複数のキーワードが含まれる場合にキーワードの優先ルールを適用し、優先順位の最も高いキーワードにより対応テーブルを参照** |

#### 侵害となる行為

**❌ 以下を実装すると侵害の可能性が高い:**

1. 領収書の取引内容（店舗名、商品名等）からキーワードを抽出
2. キーワードと勘定科目の対応テーブル（例：「タクシー」→「旅費交通費」）を参照
3. 複数キーワードがある場合に優先ルールで1つを選択
4. 選択したキーワードに対応する勘定科目を自動的に付与

**具体例:**
```
店舗名: 「JR五反田 モロゾフ」
↓
キーワード抽出: 「JR」「五反田」「モロゾフ」
↓
対応テーブル参照: JR→旅費交通費、五反田→会議費、モロゾフ→接待費
↓
優先ルール適用: モロゾフが最優先
↓
結果: 「接待費」を自動付与

→ この処理フローは特許侵害の可能性あり
```

#### 回避方法

**✅ freee vs マネーフォワード判決に基づく回避策:**

マネーフォワードが勝訴した理由は、**機械学習**を使用していたため。

| 方式 | 侵害リスク | 理由 |
|------|-----------|------|
| キーワード対応テーブル方式 | **高** | 特許請求項に該当 |
| 機械学習による推定方式 | **低** | 裁判で非侵害と認定 |
| 利用者の過去履歴から学習 | **要確認** | グレーゾーン |
| 勘定科目候補を提示し利用者が選択 | **低** | 「自動的に仕訳」に該当しない |
| 会計ソフトに構造化データのみ提供 | **なし** | 勘定科目判定を行わない |

### 2.2 freee特許5936284（機械学習を用いた自動仕訳）

#### 特許の技術的範囲

**概要:** 銀行口座・クレジットカード明細からAI/機械学習で勘定科目を推測

#### 侵害となる行為

**❌ 以下を実装すると侵害の可能性がある:**

1. 銀行口座またはクレジットカード明細を取得
2. 機械学習モデルで勘定科目を推測
3. 推測結果に基づいて自動仕訳

**⚠️ 本サービスでの注意点:**

| データソース | 侵害リスク |
|-------------|-----------|
| 銀行/クレカ明細 | **高**（特許の対象） |
| 領収書データ | **要確認**（データソースが異なる） |

#### 回避方法

**✅ 推奨アプローチ:**

1. **データソースの明確化**: 領収書データは銀行/クレカ明細とは異なるデータソース
2. **勘定科目判定を行わない**: 構造化データのみを提供し、判定は会計ソフトに委譲
3. **弁理士への確認**: 領収書データに機械学習を適用した場合の侵害リスクを確認

### 2.3 東芝テック特許6598936（電子レシートシステム）

#### 特許の技術的範囲

**主な構成要素:**

- 消費者コード（バーコード等）と電子レシート情報の対応付け
- キャンペーン情報の表示
- 店舗側端末と利用者端末間の連携

#### 侵害となる行為

**❌ 以下を実装すると侵害の可能性がある:**

1. 会計時に利用者がバーコードを提示
2. 店舗端末がバーコードを読み取り
3. バーコードと紐付いた電子レシートをサーバに送信
4. 利用者端末で電子レシートを受信・表示
5. キャンペーン情報を電子レシートと連動して表示

#### 回避の可能性

**✅ 本サービスとの差異:**

| 観点 | 東芝テック特許 | 本サービス |
|------|---------------|-----------|
| 対象文書 | **レシート**（買い物明細） | **領収書** |
| コードの表示者 | 利用者がバーコードを提示 | **店舗がQRコードを表示**（逆方向） |
| 認証方式 | 固定的な消費者コード | **ワンタイムトークン** |
| 主目的 | 家計管理・販促 | **経費精算** |

**重要な差異:** 
- 東芝テックFAQより「領収書については電子化はされず、今まで通り紙で出力されます」
- 本サービスは「領収書」の電子化が目的であり、直接競合しない

**⚠️ 要確認事項:**
- QRコードを**利用者側が表示**する方式を採用する場合は侵害リスクあり
- **店舗側がQRコードを表示**する方式であれば差異化できる可能性

---

### 2.4 東芝テック特許6618585（電子レシート・返品処理）

#### 特許の技術的範囲

**主な構成要素:**
- 返品処理時に訂正電子レシートデータを生成
- 元の電子レシートとの関連付け

#### 侵害となる行為

**❌ 以下を実装すると侵害の可能性がある:**

1. 既存の電子レシートに対する返品処理
2. 訂正レシートデータの自動生成
3. 元レシートとの紐付け管理

#### 回避方法

**✅ 推奨アプローチ:**

1. 領収書の訂正・削除は電帳法対応の「訂正削除履歴」として管理
2. 返品処理という概念ではなく、「領収書の取消・再発行」として設計
3. レシートとは異なる文書として位置づけ

---

### 2.5 QRコード関連特許（デンソーウェーブ）

#### 現在の状況

**✅ 安心して使用可能:**

- QRコードの基本特許は**権利満了**
- JIS/ISO規格に準拠したQRコードは**誰でも無料で使用可能**
- デンソーウェーブは「QRコード」の商標は保有しているが、使用料は不要

#### 注意事項

| 状況 | 対応 |
|------|------|
| 標準規格準拠のQRコード | 自由に使用可能 |
| 規格から逸脱したQRコード | 他社特許を確認する必要あり |
| 「QRコード」という名称の使用 | 商標であることを認識した上で使用 |

## 3. 網羅性の評価

### 3.1 調査済み領域

| 領域 | 調査状況 | 主要プレイヤー |
|------|----------|---------------|
| 電子レシート | ◎ 調査済 | 東芝テック |

| QRコード認証 | ◎ 調査済 | デンソーウェーブ |
| 電子帳簿保存法対応 | ○ 一般的な技術（特許少） | - |

### 3.2 追加調査が必要な領域

| 領域 | 理由 | 推奨アクション |
|------|------|---------------|
| POSベンダー各社の特許 | Square、スマレジ等が関連特許を保有している可能性 | J-PlatPatで追加調査 |
| 経費精算システム | 楽楽精算、ジンジャー経費等の特許 | J-PlatPatで追加調査 |
| 認定タイムスタンプ | 時刻認証に関する特許 | 弁理士に確認 |
| 海外特許 | 米国・欧州での類似特許 | 国際展開時に調査 |

### 3.3 公開特許出願（未登録）の監視が必要

特許出願は出願から**1年6ヶ月後**に公開されるため、最新の出願は公開されていない可能性がある。

**推奨:** 
- J-PlatPatで定期的に関連キーワードで検索
- 特許公報の公開を監視するサービスの利用を検討

## 4. 侵害回避のための設計指針

### 4.1 勘定科目判定について

**❌ 避けるべき実装:**
```
店舗名/商品名 → キーワード抽出 → 対応テーブル参照 → 勘定科目自動付与
```

**✅ 推奨する実装:**

| オプション | 方式 | 侵害リスク |
|-----------|------|-----------|
| A | 勘定科目判定を行わない（構造化データのみ提供） | **なし** |
| B | 利用者が手動で勘定科目を選択 | **なし** |
| C | 候補を提示し利用者が確定（自動仕訳ではない） | **低** |
| D | 会計ソフト連携時に会計ソフト側で判定 | **なし** |

### 4.2 QRコード認証フローについて

**❌ 避けるべきフロー:**
```
利用者がQRコード/バーコードを表示 → 店舗が読み取り → 電子レシート送信
```

**✅ 推奨するフロー:**
```
店舗がQRコードを表示 → 利用者が読み取り → ワンタイムトークンで認証 → 電子領収書発行
```

**差異化ポイント:**
1. QRコードの表示者が逆（店舗側が表示）
2. ワンタイムトークンによる認証
3. 対象文書が「領収書」（レシートではない）

### 4.3 電帳法対応について

**✅ 特許リスクが低い実装:**

- 訂正削除履歴の管理
- タイムスタンプ付与（認定タイムスタンプ事業者のサービスを利用）
- 検索要件の充足

**理由:** 電帳法対応は法的要件であり、特定の実装方法が特許で保護されているケースは少ない

## 7. 結論

### 7.1 主要リスクまとめ

| リスク領域 | レベル | 対策 |
|-----------|--------|------|
| 勘定科目自動判定 | **高** | 実装しない、または会計ソフト連携 |
| 電子レシート類似フロー | **中** | 「領収書」として差別化、フロー設計を変更 |
| QRコード使用 | **低** | 標準規格準拠で問題なし |
| 電帳法対応 | **低** | 一般的技術で対応可能 |

### 7.2 調査の網羅性

- **現状**: 主要な競合（東芝テック、freee）の特許は調査済み
- **追加調査必要**: POSベンダー各社、経費精算システムベンダーの特許
- **継続監視必要**: 新規公開特許、競合の出願動向

### 7.3 最終推奨

1. **勘定科目自動判定は実装しない**（または会計ソフト連携のみ）
2. **「電子領収書」として明確に差別化**（レシートではない）
3. **弁理士による専門的調査を実施**
4. **電子領収書発行フローの自社特許出願を検討**
