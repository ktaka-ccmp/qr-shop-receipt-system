# 電子領収書発行システム ビジネスプラン

## Executive Summary

**サービス名称（仮）:** e-Receipt / デジタル領収書

**コンセプト:** 店舗が発行する領収書を電子化し、経費精算するビジネスパーソンに直接届けるB2B2Cプラットフォーム

**核心的価値:**
- OCR不要で100%正確な構造化データ
- 電子帳簿保存法への自動対応
- 印紙税の削減（5万円超の領収書で200円→0円）
- 領収書の紛失リスク排除

**ターゲット市場:** 経費精算を行うビジネスパーソン × 領収書を発行する店舗（飲食店、小売店等）

---

## 1. サービス概要

### 1.1 解決する課題

| ステークホルダー | 現状の課題 |
|-----------------|-----------|
| **経費精算者** | 領収書の紛失、OCRの誤認識、電帳法対応の手間 |
| **店舗** | 印紙税コスト（5万円超で200円〜）、紙・印刷コスト |
| **経理担当者** | 領収書の突合作業、電帳法対応の運用負荷 |

### 1.2 サービスの仕組み

```
┌─────────┐    ┌─────────────┐    ┌─────────┐
│  店舗   │───▶│ 本サービス  │◀───│ 利用者  │
│ (POS)  │    │  サーバー   │    │ (アプリ) │
└─────────┘    └─────────────┘    └─────────┘
     │                │                │
     │ ① 決済完了     │                │
     │ ② 領収書データ送信              │
     │                │                │
     │ ③ QRコード表示  │                │
     │ ─────────────────────────────▶ │
     │                │                │
     │                │ ④ QR読取・参照  │
     │                │◀───────────────│
     │                │                │
     │                │ ⑤ 電子領収書配信 │
     │                │───────────────▶│
```

### 1.3 利用フロー

**店舗側:**
1. 通常通り決済処理
2. 利用者が電子領収書を希望 → POSで「電子領収書」ボタン
3. QRコードを表示（またはレシートに印刷）

**利用者側:**
1. アプリでQRコードをスキャン
2. 電子領収書を受信・保存
3. 必要に応じて仕訳CSV出力、会計ソフト連携

## 2. 市場分析

### 2.1 市場規模

| 指標 | 数値 | 出典 |
|------|------|------|
| 国内の経費精算市場 | 約500億円（2024年） | 各種調査 |
| 電子帳簿保存法対応ニーズ | 急拡大中 | 2024年1月義務化 |
| 印紙税の年間税収 | 約3,000億円 | 財務省 |

### 2.2 競合分析

| サービス | 対象 | 特徴 | 本サービスとの違い |
|----------|------|------|-------------------|
| **スマートレシート（東芝テック）** | レシート | 17,000店舗、家計管理向け | **領収書は対象外** |
| **楽楽精算、TOKIUM等** | 経費精算 | OCRで領収書読取 | **店舗発行データではない** |
| **freee、マネーフォワード** | 会計全般 | 自動仕訳機能 | **領収書発行機能なし** |

### 2.3 差別化ポイント

| 観点 | 既存サービス | 本サービス |
|------|-------------|-----------|
| **データ精度** | OCR（90〜95%） | **100%（店舗発行データ）** |
| **対象文書** | レシート or 撮影画像 | **領収書（電子原本）** |
| **電帳法対応** | 利用者側で対応 | **自動対応** |
| **印紙税** | 紙なら必要 | **不要** |

## 3. ビジネスモデル

### 3.1 B2B2Cモデル

```
本サービス ──▶ POSベンダー ──▶ 店舗 ──▶ 利用者
  (API)        (再販)        (発行)    (受領)
```

**POSベンダー経由の理由:**
- 営業効率（1社契約で数百〜数千店舗に展開）
- 技術統合の容易さ（POSベンダーがAPI実装）
- 店舗の導入障壁低減

### 3.2 料金設計

#### 店舗側（POSベンダー経由）

| 項目 | 金額 | 備考 |
|------|------|------|
| **卸値（本サービス→POSベンダー）** | 10円/枚 | |
| **販売価格（POSベンダー→店舗）** | 15〜30円/枚 | ベンダー裁量 |
| **店舗のメリット** | 印紙税200〜400円 vs 15〜30円 | 10倍以上の削減 |

#### 利用者側（フリーミアム）

| プラン | 月額 | 機能 |
|--------|------|------|
| **Free** | 0円 | 領収書受信・閲覧（3ヶ月保存） |
| **Light** | 200円 | 全期間保存、PDF出力 |
| **Standard** | 500円 | 仕訳CSV出力、会計ソフト連携 |

### 3.3 収益試算

**前提条件:**
- 導入店舗: 1,000店舗
- 月間発行枚数: 100枚/店舗
- 利用者の有料化率: 20%
- 有料プラン平均単価: 350円/月

| 収益源 | 計算 | 月間収益 |
|--------|------|----------|
| **店舗側** | 1,000店 × 100枚 × 10円 | 100万円 |
| **利用者側** | 100,000枚 × 20% × 350円 ÷ 10枚 | 70万円 |
| **合計** | | **170万円/月** |

**スケール時（10,000店舗）:** 約1,700万円/月

## 4. 技術設計

### 4.1 技術スタック（想定）

| レイヤー | 技術 | 理由 |
|----------|------|------|
| **バックエンド** | Rust / Axum | 高性能、型安全 |
| **データベース** | PostgreSQL | 電帳法対応の履歴管理 |
| **認証** | OAuth2 / WebAuthn | セキュリティ、既存ライブラリ活用 |
| **インフラ** | GCP / Kubernetes | スケーラビリティ |
| **API** | REST / GraphQL | POSベンダー連携 |

### 4.2 電子帳簿保存法対応

| 要件 | 対応方法 |
|------|----------|
| **真実性の確保** | 訂正削除履歴の自動記録 |
| **可視性の確保** | 検索機能（日付・金額・取引先） |
| **タイムスタンプ** | 方法B（訂正削除履歴）→ 方法A（認定タイムスタンプ）へ段階実装 |

### 4.3 QRコード＋ワンタイムトークン

```
QRコード内容: https://receipt.example.com/r/{one_time_token}

- one_time_token: 有効期限付き（例: 24時間）
- 1回の読取で利用者アカウントに紐付け
- 紐付け後はトークン無効化
```

## 6. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| **POSベンダーの採用意欲が低い** | 高 | 導入店舗の満足度向上をデータで示す、パートナープログラム設計 |
| **東芝テックが領収書機能を追加** | 高 | 先行して市場獲得、経費精算特化で差別化 |
| **利用者の有料化が進まない** | 中 | 無料機能を制限、法人向けプラン追加 |
| **電帳法の要件変更** | 中 | 法改正動向のモニタリング、柔軟な設計 |
| **鶏と卵問題（店舗と利用者）** | 高 | POSベンダー経由で店舗を先に確保 |
